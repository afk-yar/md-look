## Подробно

### T-01. Дедупликация окон + foreground при открытии из проводника `[S]`

**Проблема:** Два связанных бага при открытии файла через проводник:
1. При двойном клике на файл, который уже открыт в MDLook, создаётся новое окно с тем же файлом вместо активации существующего.
2. При открытии нового файла окно часто оказывается на заднем плане — приходится кликать на иконку в панели задач.

**Корень:** В IPC-обработчике (`app.py:150-153`) команда `OPEN:` безусловно вызывает `_create_window(filepath)` без проверки дублей и без вызова `_force_foreground()`:
```python
if data.startswith('OPEN:'):
    filepath = data[5:]
    if os.path.isfile(filepath):
        _create_window(filepath)  # нет проверки дублей, нет foreground
```
При этом `_force_foreground()` уже реализован (`app.py:48-98`) и вызывается для команды `SHOW`, но не для `OPEN:`.

**Фикс:** В обработчике `OPEN:` перед `_create_window`:
1. Нормализовать путь (`os.path.normcase`) для case-insensitive сравнения.
2. Пройти по `_windows` и сравнить `os.path.normcase(entry['api'].current_path)` с нормализованным путём (normcase на обеих сторонах, т.к. `current_path` хранится без нормализации).
3. Если найден — вывести конкретное окно на передний план (по HWND этого окна, а не всех; `_force_foreground()` поднимает ВСЕ окна — не подходит). Если окно скрыто в трей — сначала `window.show()`.
4. Если не найден — `_create_window(filepath)` + foreground нового окна. Учесть, что `_create_window` из IPC-потока ставит запрос в очередь GUI-потока — HWND может не существовать сразу; использовать callback `on_loaded` или аналог для foreground.

**Definition of Done:**
- [ ] Двойной клик на файл, уже открытый в MDLook, активирует существующее окно (не создаёт новое)
- [ ] Работает при разном регистре путей (`C:\Docs\file.md` vs `c:\docs\File.md`)
- [ ] Если существующее окно свёрнуто в трей — восстанавливается и получает фокус
- [ ] При открытии нового файла окно появляется на переднем плане с фокусом
- [ ] Если файл НЕ открыт — открывается как обычно

### T-02. Поиск по странице через Ctrl+F `[M]`

**Проблема:** Нет механизма поиска текста на отрендеренной странице. Пользователь ожидает стандартное поведение Ctrl+F.

**Подход:** Исследовать два варианта, выбрать один:
- **Встроенный WebView2 find** — `ICoreWebView2` имеет find API, но pywebview может не экспонировать его. Если доступен — предпочтительнее (нативное поведение, поиск в скрытом контенте, перформанс).
- **Кастомный JS поиск-бар** — реализация через DOM API (window.find / TreeWalker / CSS Highlight API). Полный контроль над UI, но сложнее и есть edge cases. **Важно:** при JS-варианте нужно подавить нативный Ctrl+F WebView2 (через AcceleratorKeyPressed или аналог в pywebview), иначе откроются два поисковых бара одновременно.

**Фикс:** Реализовать поиск (появляется по Ctrl+F), который ищет текст на отрендеренной странице. Поддержать: подсветку совпадений, переход между ними (Enter / Shift+Enter или стрелки), счётчик совпадений, закрытие по Escape.

**Definition of Done:**
- [ ] Ctrl+F открывает поисковую строку
- [ ] Введённый текст подсвечивается на странице
- [ ] Переход между совпадениями (вперёд/назад)
- [ ] Счётчик совпадений (N из M)
- [ ] Escape закрывает поиск
